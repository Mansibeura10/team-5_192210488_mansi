⬡ PROBLEM STATEMENT

Telecom companies generate a massive volume of Call Detail Records (CDRs) every day. Each record contains who made the call, who received it, call duration, call type, tower used, and timestamp. The data is raw, messy, and needs automated processing.

The goal is to build a robust ETL pipeline using Informatica that reads daily CDR CSV files, cleans and transforms them, calculates revenue and call classification fields, and loads the results into database staging and final tables for analysis.

⬡ CORE ETL PIPELINE — STEP BY STEP WORKFLOW
EXTRACT
STEP 01 · EXTRACT

Load CDR CSV File

Informatica reads the daily CDR flat file from the source storage location using a Flat File Source Definition.

  • Connect to the source folder containing CDR_RAW_Telecom.csv
  • Define source schema: CallID, Caller, Receiver, Duration, CallType, TowerID, Timestamp
  • Also load Customer_Master and Tower_Master as lookup sources
  • Schedule to trigger daily (batch mode)
Informatica Source Qualifier
STEP 02 · EXTRACT

Source Qualifier & Initial Pass

Source Qualifier transformation reads and passes raw rows into the pipeline for downstream processing.

  • All 7 columns from CDR are passed through
  • Customer and Tower master data loaded as lookup tables
  • No filtering at this stage — all rows come in
Informatica Workflow Manager
⬇
TRANSFORM
STEP 03 · TRANSFORM

Duplicate Removal

Remove duplicate CDR entries to ensure data accuracy and prevent double-counting in reports.

  • Use Sorter Transformation on CallID + Timestamp
  • Apply Aggregator or Router to identify and discard duplicate rows
  • Keep only one record per unique CallID
Sorter + Router Transformation
STEP 04 · TRANSFORM

Date / Time Standardization

Fix incorrect or inconsistent date/time formats in the Timestamp column to a unified standard.

  • Use Expression Transformation with TO_DATE() function
  • Convert all formats → YYYY-MM-DD HH24:MI:SS
  • Flag rows with unparseable timestamps for audit log
Expression Transformation
STEP 05 · TRANSFORM

Null Duration Handling

Replace missing or null call duration values with 0 to prevent calculation errors downstream.

  • Use Expression Transformation: IIF(ISNULL(Duration), 0, Duration)
  • Ensures Revenue calculation doesn't fail on NULLs
IIF(ISNULL(Duration), 0, Duration)
STEP 06 · TRANSFORM

Data Validation Checks

Add validation to catch bad data — invalid phone numbers, negative durations, malformed records.

  • Check Duration ≥ 0 (reject negatives → reject file)
  • Validate Caller / Receiver phone number format
  • Route bad records to error/reject pipeline
  • Capture validation failure counts in audit log
Router Transformation · Reject Port
STEP 07 · TRANSFORM · CALCULATED FIELDS

Revenue Calculation

Compute the revenue generated from each call based on duration and a per-second charge rate.

FORMULA
Revenue = Duration × 0.02
  • Charge rate: ₹0.02 per second
  • Implemented in Expression Transformation
  • Output: new column Revenue added to the pipeline
Expression: Duration * 0.02
STEP 08 · TRANSFORM · CALCULATED FIELDS

IsInternational Flag

Classify each call as international or domestic based on CallType field.

FORMULA
IsInternational = IIF(CallType = 'INT', 1, 0)
  • CallType values: LOCAL, STD, INT
  • INT → IsInternational = 1
  • LOCAL / STD → IsInternational = 0
  • Enables easy international call filtering in analytics
IIF(CallType = 'INT', 1, 0)
STEP 09 · TRANSFORM · ENRICHMENT

Customer & Tower Lookup

Enrich CDR records by joining with Customer Master and Tower Master using Lookup transformations.

  • Lookup on Caller phone → get CustomerName, PlanType
  • Lookup on TowerID → get Region, City
  • Use Lookup Transformation (connected, cached)
  • Unmatched records flagged in audit log
Lookup Transformation (Cached)
⬇
LOAD
STEP 10 · LOAD

Load to Staging Table

Load cleaned, transformed CDR data into the database Staging Table as a holding area before final load.

  • Target: STG_CDR_Telecom (staging table in MySQL/Oracle)
  • Truncate-and-Insert mode (daily refresh)
  • All transformed columns loaded including Revenue & IsInternational
  • Staging table retains data for audit and re-processing
Informatica Target Definition · Staging
STEP 11 · LOAD

Load to Final Table

Move validated, enriched data from staging into the Final Production Table for reporting and analytics.

  • Target: CDR_FINAL_Telecom (production table)
  • Insert or Upsert based on CallID (primary key)
  • Includes CustomerName, Region, City from lookups
  • This table feeds all analytical use cases and dashboards
Informatica Target Definition · Final
STEP 12 · LOAD · AUDIT

Capture Load Audit Logs

Record success/failure counts, row counts, and error details for monitoring and debugging.

  • Log: Total rows read, rows inserted, rows rejected
  • Capture timestamp of each load run
  • Write to AUDIT_LOG table in database
Workflow Monitor · Session Log

High Usage Alert Table

Detect customers with unusually high call duration and flag them in a dedicated alert table.
